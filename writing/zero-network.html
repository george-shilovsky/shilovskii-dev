Zero network: how Migaja achieves privacy by construction

Most "private" apps achieve privacy with a policy. Migaja achieves it with the binary — the app cannot phone home because macOS will not let it open a socket.

Most "private" apps achieve privacy with a policy. Migaja achieves it with the binary. The app cannot phone home because macOS will not let it open a socket.

The entitlements file

The entitlements that ship inside the .app bundle do not contain com.apple.security.network.client or com.apple.security.network.server. App Sandbox is on. Hardened runtime is on. The kernel enforces this: any connect(2) call from the app process is denied by the sandbox.

That is the load-bearing claim. Everything below is consistent with it, but the entitlements alone are what make this provable. You can inspect them yourself with codesign -d --entitlements - /Applications/Migaja.app after you install the app.

The model is bundled, not downloaded

The 300 MB embedding model ships inside the .app. There is no downloader, no fallback CDN, no first-run network step. The model loader uses Bundle.main to find the file by resource name — exactly the same code path Apple uses to load any other bundled asset. If you yank your Wi-Fi after install, semantic search continues to work.

No third-party SDKs

There is no Sentry, no Crashlytics, no Firebase, no Amplitude, no Segment, no Sparkle, no auto-updater, no analytics. The PrivacyInfo.xcprivacy shipped in the bundle declares:

NSPrivacyTracking = false
NSPrivacyCollectedDataTypes = []

This matches the App Privacy Nutrition Label on the App Store listing: Data Not Collected.

Filtering secrets out of the pasteboard

Three pasteboard hints are dropped before a clip is ever written to disk:

  • org.nspasteboard.TransientType — used by tools that copy intermediate values they don't want recorded.
  • org.nspasteboard.ConcealedType — used by password managers for short-lived secrets.
  • com.apple.is-remote-clipboard — handoff content from another device.

On top of that, Migaja ships with a default app-exclusion list: 1Password, Bitwarden, and Keychain Access. Anything you copy from those apps is ignored at the source — it never enters the database, never gets embedded, never gets indexed. You can edit the list in Settings → Exclusions.

Soft delete with a 24-hour escape hatch

When you delete a clip, it is first marked with a deletion timestamp and hidden from the UI. A background sweep hard-deletes it 24 hours later. If you change your mind in that window, you can restore it. After the window, the row is gone from SQLite for good.

Where your data lives

  • Clips and embeddings: a SQLite database inside Migaja's sandboxed Application Support container.
  • Trial timestamp: the macOS Keychain, so you can't reset the trial by reinstalling.
  • Preferences: UserDefaults.

Nothing leaves the disk.

The one network call: StoreKit 2

The only outbound traffic ever initiated on Migaja's behalf is StoreKit 2 — when you start a subscription, restore one, or your Mac periodically refreshes the receipt. That traffic is mediated entirely by macOS and the App Store. Migaja itself does not open the socket; it asks StoreKit to talk to Apple, and the kernel allows that because it's Apple's framework, not the app's. There are no other endpoints.

Zero TCC permissions, by design

Migaja never asks for Accessibility, Full Disk Access, Screen Recording, Automation, or any other macOS permission. When you pick a clip, it writes the text to the system clipboard (NSPasteboard, a public sharing mechanism) and gives focus back to the app you came from — you press ⌘V yourself. This is one extra keystroke vs. an auto-pasting clipboard manager, and in exchange the app physically cannot read other apps' UI, scrape passwords, or simulate input. The binary doesn't contain the code paths that would let it.

This is what "privacy by construction" actually means: not a promise on a marketing page, but a build that physically cannot do the things it promises not to do.
