How Migaja searches your clipboard offline

Two retrieval stages, one query box, zero network. A walk through the pipeline that turns "the postgres connection string from last week" into the right clip in under 300 ms.

Migaja is a macOS clipboard manager. When you press ⌘⇧V, you can type any phrase — "the postgres connection string from last week", or "that dark-red hex I picked" — and find the clip without remembering the exact words. The search runs entirely on your Mac. There is no server.

The pipeline has two stages, behind the same query box.

Phase 1 — full-text search in well under 5 ms

Every clip is captured from NSPasteboard (1.0-second polling) and persisted in SQLite via GRDB. Alongside the main clips table, Migaja maintains an FTS5 virtual table mirrored by triggers, so inserts, updates, and deletes stay in lockstep with the index without any manual maintenance.

The tokenizer is unicode61 remove_diacritics 2. That single config decision is what lets you find a clip you copied as café by typing cafe, or hit Größe by typing grosse — useful in a multilingual workflow.

When you start typing, Migaja issues an FTS5 MATCH query and keeps the result order but discards the raw BM25 scores. Only the relative rank is used for the next step.

Phase 2 — semantic search in 50–300 ms

The embedding model is multilingual-e5-base-q8_0.gguf — 768 dimensions, Q8_0 quantisation, ~300 MB on disk, ~100 languages, 512-token context window. It ships inside the .app. There is no first-run download.

Inference runs on llama.cpp, statically linked into the app, with Metal as the backend. Pooling is mean, and the resulting vector is L2-normalised before it ever touches the database.

e5 is task-prefixed, so Migaja prepends the prefixes itself: "query: " for what you type, "passage: " for clip text. (A common foot-gun when wiring e5 is double-prefixing from both the caller and the engine; Migaja asserts against it.)

Vectors are stored on each row as a BLOB — little-endian IEEE-754 float32, length embedding_dim * 4 bytes. Because every vector is L2-normalised, cosine similarity collapses to a dot product. Migaja uses Apple's Accelerate framework — vDSP_dotpr against the raw blob bytes — so there's no per-row allocation and no per-query sqrt.

Hybrid ranking

The final score for a clip is:

final = (1 − semWeight) * textScore + semWeight * cosine

semWeight defaults to 0.6 — a deliberate tilt toward semantic relevance, on the theory that if you remembered the exact words you'd just press ⌘V. Pinned clips always sort above the rest, via a stable predicate that runs before the score comparison.

FTS results render immediately. Semantic results merge in with a short animation when phase 2 finishes. The popup never blocks on the embedding pass, which keeps the experience snappy even on a cold launch.

Lifecycle

The embedding model only stays resident while you're actually searching. After 30 seconds of inactivity it is unloaded and roughly 350 MB of RAM is returned to the system. The next query reloads it transparently.

Why static llama.cpp

The Mac App Store sandbox forbids posix_spawn of arbitrary binaries, so Migaja can't shell out to a separate inference process. Instead, the llama.cpp and ggml libraries (libllama.a, libggml.a, the Metal/BLAS/CPU backends) are linked directly into the app. The C API is bridged into Swift via a single bridging header.

The net result.
768-dimensional multilingual semantic search, on your laptop, with no network round-trip and no per-query RAM cost when you're idle.
